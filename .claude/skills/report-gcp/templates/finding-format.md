# 發現（Finding）統一格式

所有分析 agent（security-auditor、reliability-reviewer、performance-reviewer、cost-optimizer、
ops-reviewer）輸出 `findings/<pillar>.md` 時，必須遵守本格式，report-writer 依此彙整。

## 檔案結構

```markdown
# <支柱名稱>分析結果

- 分析日期：YYYY-MM-DD
- 資料來源：data/ 掃描於 <掃描時間，取自 data/scan-meta.json>
- 支柱評分：<1-5 分，5 為最佳>
- 發現統計：高 X 項｜中 Y 項｜低 Z 項

## 發現清單

### [<嚴重度>] <發現標題>

- **編號**：<支柱代碼>-<流水號>（如 SEC-01、REL-01、PERF-01、COST-01、OPS-01）
- **嚴重度**：高｜中｜低
- **受影響資源**：<資源名稱 / self link / 執行個體名，多個用清單>
- **現況證據**：<描述現況，並註明依據哪個 data/ 檔案的哪個欄位>
- **風險/影響**：<不處理會發生什麼>
- **建議做法**：<具體可執行的改善步驟>
- **參考文件**：<Google Cloud 官方文件連結，至少一個>
- **預估工作量**：小（<1 天）｜中（1-5 天）｜大（>5 天）

## 良好實務（已符合項目）

- <已經做對的事項，報告中用來平衡呈現>

## 資料缺口

- <因權限不足、API 未啟用或未掃描而無法判斷的項目>
```

## 嚴重度定義

| 嚴重度 | 定義 |
|---|---|
| 高 | 有直接安全風險、單點故障、或每月浪費顯著成本，應立即處理 |
| 中 | 偏離最佳實務，中期內應排入改善 |
| 低 | 優化建議，有餘裕時處理 |

## 規則

1. 每項發現的「現況證據」必須能對回 `data/` 內的實際掃描資料，不得憑空推測
2. 「參考文件」優先引用 Google Cloud 官方來源：Well-Architected Framework、服務文件、
   Architecture Center、安全基礎藍圖
3. 無法確認的項目寫進「資料缺口」，不要當成發現
4. 若某類資源不存在（例如專案沒有 Cloud SQL），直接略過，不需產生發現
5. **「掃描回空清單」與「查詢失敗」意義相反**，不可混用：
   前者是有效證據（該項確實未設定），後者是資料缺口。判定一律以
   `data/digest/scan-gaps.md` 為準
